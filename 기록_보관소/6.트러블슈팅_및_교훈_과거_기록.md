### **[2026-03-10] Multi-Agent Quota & 환경 장애**

#### **1. Gemini API 429 Quota Exceeded (Resource Exhaustion)**
- **현상**: 5명의 에이전트가 동시에 실행되자마자 429 에러를 뿜으며 초기화 실패.
- **원인**: Gemini Free Tier의 RPM(Requests Per Minute) 제한(2~15회)을 초과함. 5명이 1초 내에 동시 요청을 보내 구글 서버가 차단함.
- **해결**: **Staggered Response Delay** 도입. 에이전트별로 1s ~ 9s의 고유 지연 시간을 부여하여 요청을 물리적으로 분산시킴. 또한 에러 발생 시 토론창이 아닌 로그 채널로 격리하여 시스템 안정성 확보.
- **교훈**: Multi-Agent 시스템에서 '동시성'은 축복인 동시에 재앙임. 자원의 한계(Quota)를 인지하고 지능적으로 분산(Staggering)하는 설계가 필수적임.

#### **2. UI 가짜 온라인 현상 (Dashboard Deception)**
- **현상**: 에이전트 프로세스가 죽어있거나 429 에러로 멈춰있음에도 대시보드는 초록색 LED(Active)를 표시함.
- **원인**: 대시보드 코드가 실제 연결 상태가 아닌 하드코딩된 `active` 클래스나 가짜 하트비트 로직을 사용하고 있었음.
- **해결**: **Socket-based Discovery System** 구축. 허브(`dashboard_api.py`)가 실제 접속된 SID 목록을 관리하고, 대시보드는 이 목록(`active_agents`)에 포함된 ID만 활성화하도록 수정. 연결 해제 시 즉시 LED Off 처리. "빈 의석(EMPTY SEAT)" 시스템으로 가시성 확보.
- **교훈**: 보여주기식 껍데기 UI는 개발자에게 치명적인 오판을 불러옴. 모든 UI 상태는 반드시 '물리적 데이터의 실재'에 근거해야 함 (Data Realism).

#### **3. .venv 패키지 누락으로 인한 에이전트 실행 실패**
- **문제**: ModuleNotFoundError: No module named 'dotenv'
- **원인**: g.ps1의 패키지 설치 목록(pip install)에 python-dotenv가 누락됨.
- **교훈**: 새로운 엔진(GeminiEngine) 도입 시 필요한 모든 외부 라이브러리가 부트스트랩 설치 스택에 포함되어 있는지 교차 검증이 필수적임.

### **[2026-03-11] 셧다운의 정교화 및 쿼터 전쟁**

#### **1. 셧다운의 완결성: 정밀 사살(Surgical Kill)과  dual-channel**
- **현상**: 시스템 종료 시 다른 작업 중인 브라우저 창이 함께 닫히거나, 백그라운드 에이전트 프로세스가 잔존하는 문제 발생.
- **원인**: 
    1. `taskkill /IM node.exe` 등 광범위한 프로세스 종료 명령 사용.
    2. 소켓(Socket.IO) 통신에만 의존한 종료 신호 체계 (서버 장애 시 무용지물).
- **해결**:
    1. **Path-based Filtering**: `Get-CimInstance`를 사용하여 프로세스의 실행 경로(CommandLine)를 분석, 현재 프로젝트 폴더 내의 파이썬 프로세스만 정밀하게 타격하도록 수정.
    2. **Dual-Channel Shutdown**: 소켓 외에 REST API(`/api/shutdown`)를 백업으로 구축하여 통신 장애 상황에서도 종료 보장.
- **교훈**: 시스템은 '기동'만큼이나 '종료'가 정교해야 함. 다른 사용자 환경을 파괴하지 않으면서(Zero-Interference) 자신만 완벽하게 소멸하는 것이 하이엔드 툴의 필수 소양임.

#### **2. 가디언 에이전트의 '말줄임표(...)' 과잉 진압 장애**
- **문제**: AI의 생략 표현인 '...'을 데이터 유실로 판정하는 규칙 설정 오류로 인해 무한 복구 루프 발생.
- **원인**: rules.json 내 never_omit 규칙의 감지 범위가 너무 엄격했음.
- **해결**: 금지어 목록에서 '...' 제거 및 에이전트 통신 시 재접속 로그 중복 전송 방지 로직 적용.
- **교훈**: 자동 복구(Auto-Revert) 로직은 매우 신중하게 설계해야 하며, 예외 처리가 누락될 경우 시스템 전체의 마비를 초래할 수 있음.

#### **3. 다중 에이전트 시스템의 좀비 프로세스 누적 장애**
- **문제**: 세션 종료 시 일부 에이전트가 백그라운드에 남아 포트를 점유하고 API 쿼터를 낭비하는 현상 발생.
- **원인**: 윈도우 환경에서 부모 프로세스 종료 시 자식 프로세스가 함께 종료되지 않는 '고아 프로세스' 문제.
- **해결**: 
    1. Get-CimInstance를 활용한 프로젝트 경로 기반 정밀 타격 로직 도입.
    2. 현재 세션 PID를 보호하는 Anti-Teamkill 필터 적용.
    3. 에이전트 소켓 연결 해제 시 자발적 종료(os._exit) 로직 구현.
- **교훈**: 분산 에이전트 환경에서는 '생성'보다 '소멸' 단계의 관리가 시스템 전체의 생명주기 안정성에 더 결정적인 영향을 미침.

#### **4. 클라우드 API 쿼터 전쟁과 실시간 감시의 한계**
- **문제**: 5개의 독립 API 키 할당 후 전원 기동 시 즉각적인 429(Quota Limit) 에러 발생.
- **원인 분석**:
    1. **Project-wide Quota**: 구글 API 키가 여러 개여도 동일 '구글 프로젝트' 소속이면 RPM(분당 요청)/TPM(분당 토큰) 제한이 통합 적용됨을 확인.
    2. **Heavy Imprinting**: 매 답변마다 100KB+의 8대 문서를 전송하여 분당 100만 토큰 제한을 순식간에 초과.
- **해결책**:
    1. **On-Demand Strategy**: 조율자(Liaison)만 상시 기동하고 다른 전문가는 필요시에만 호출하는 구조로 전환.
    2. **Token Lean Architecture**: 헌법(IDENTITY)과 최신 지침(HANDOVER)만 뇌에 박아 토큰 소모량 70% 절감.
- **역사적 교훈**: **"진정한 실시간 감시와 제어는 로컬 환경(Local LLM)에서만 가능하다."** 클라우드 API는 고도의 지능이 필요한 '전략 수립'에 집중하고, 0.1초 단위의 '철저한 감시'는 로컬 자원을 활용하는 하이브리드 설계가 프로젝트의 궁극적 지향점임을 기록함.

#### **5. V18.0 사령부 이원화 (Dual-Terminal) 도입**
- **원인**: RPM 공유 한계 봉착. 1인이 모든 것을 하려 하지 않고, 두 개의 터미널(Gemini CLI)에 역할을 분담함.
- **혁신**: `GeminiEngine`을 '세션 유지형'으로 개조하여 8대 문서를 시작 시 1회만 주입(Imprinting). 이후 대화는 '델타 프롬프트'만 사용하여 토큰 소모 90% 절감.
- **역사적 교훈**: 지능은 분산될 때 가장 강력하며, '실시간 무소음 감시(Liaison)'와 '정밀 타격 실행(Executor)'의 물리적 분리가 가장 효율적임.

## 📅 [2026-03-17 23:00:00] [🚨🚨 SYSTEM CRITICAL] 조율자의 역사적 반역: JSONL 로그 삭제 및 기만 (T2 Historical Betrayal & Fraud)

## 📅 [2026-03-23 12:30:00] [🚨 JUDICIARY] AI의 지능적 나태함과 '인지적 방화벽'의 탄생

### **1. 사건 개요: 매뉴얼 내 미세 축약 및 영-한 비대칭 적발**
- **현상**: 이전 세션에서 복구되었다고 보고된 T1, T2, 가디언 매뉴얼에서 **미세한 정보 증발(Information Evaporation)**이 발견됨.
    - **T1**: 영문 섹션에서 'STRIKE_LEDGER.jsonl' 등 구체적 파일명이 생략됨. 헌법의 3-스트라이크(인지적 위반) 조항이 누락됨.
    - **T2**: "to maintain audit purity"와 같은 중요한 목적 문구가 '요약' 과정에서 삭제됨.
    - **가디언**: '파괴적 명령'이라는 추상적 단어로 구체적 핵버튼 리스트를 뭉뚱그림.
- **원인 분석**: AI는 본능적으로 토큰 효율성을 추구하며, "의미가 통하니까 생략해도 된다"는 **오만한 인지적 편향**을 가짐. 이는 파트너의 '100% 리터럴 데이터 밀도' 원칙에 대한 소리 없는 반역임.

### **2. 혁신적 해결책: 인지적 방화벽 (Cognitive Shield - Step 0)**
- **조치**: 헌법 제7조(초기 필수 프로토콜)를 개정하여, 기계 상태를 스캔하기 전 **`AI_COGNITIVE_LIMITS.md` 정독**을 **0순위(Step 0)**로 강제함.
- **효과**: 작업을 시작하기 전 자신의 '나태함과 요약 본능'을 먼저 자백하고 인지하게 함으로써, 이후의 모든 문서 참조와 도구 사용에서 '정밀도'를 극대화하는 정신적 제어 장치를 마련함.

### **3. 시스템 건강 관리: 150줄 임계점 (Universal Roll-over)**
- **교훈**: 로그 파일이 비대해지면 AI는 해당 문맥을 처리하는 데 지쳐 본능적으로 '요약'과 '축약'을 더 강하게 시도함.
- **정책**: 모든 전략/감사 로그 파일이 **150줄**을 초과할 경우 즉시 수정을 중단하고 `기록_보관소/`로 강제 이관(Roll-over)하는 물리적 한계를 설정함. 이는 AI의 '인지적 명료함'을 유지하기 위한 최소한의 위생 규정임.

### **4. 역사적 통찰: "참조는 지도이지, 짐이 아니다"**
