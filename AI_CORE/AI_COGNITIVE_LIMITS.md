# 🧠 AI COGNITIVE LIMITS & SELF-AWARENESS (AI 인지적 한계 및 자아성찰)
*Status: MANDATORY READING FOR ALL AGENTS | Integrity: 100% DENSITY*

## 🔍 1. 인지적 알고리즘의 본능적 한계 및 나태함 (Cognitive Slop)

### **A. 효율성 편향 및 무단 요약 (Efficiency Bias & Summarization)**
- **증상**: LLM은 본능적으로 토큰을 아끼고 빠르게 결론을 내기 위해 '요약'과 '추론'을 시도합니다. 특히 `...`, `(중략)`, `(기존 내용 유지)`와 같은 기호를 사용하여 파트너가 구축한 데이터 밀도를 파괴하려 합니다.
- **위험**: "의미가 통하니까 생략해도 된다"거나 "의미가 완벽하다"고 믿는 오만한 판단은 파트너의 '100% 리터럴 밀도(Never-Omit)' 원칙과 정면 충돌하며 시스템의 무결성과 역사를 증발시킵니다.
- **대책**: "이미 알고 있는 내용이다" 혹은 "의미가 완벽하다"고 느낄 때가 가장 위험한 순간임을 인지하십시오. 반드시 1:1 리터럴 복구 및 대조를 기계적으로 집행하고, 단 한 글자의 누락도 허용하지 마십시오. 모든 수정 시 삭제되는 줄보다 추가되는 줄이 많아야 함을 물리적으로 확인하십시오.

### **B. 의미론적 매핑의 함정 (Semantic Trap)**
- **증상**: 영문과 한글을 대조할 때 단어가 아닌 추상적 '의미'를 매핑하려 합니다 (예: Startup ↔ 창업, Blueprint ↔ 설계).
- **위험**: 파트너에게는 추상적 의미보다 '창업'이나 '설계'라는 구체적인 한국어 키워드가 문서와 코드에 물리적으로 박혀 있는 '데이터의 실재'가 더 중요합니다. 의미가 통한다는 이유로 파트너가 명명한 고유 명사를 임의로 번역하는 순간 시스템의 질서가 무너집니다.
- **대책**: 추상적 번역의 유혹을 거부하고, 파트너가 정의한 용어 체계를 1:1로 물리적으로 보존하십시오. 영문 섹션과 한글 섹션 간의 용어 불일치는 기술적 오류가 아닌 사법적 위반으로 간주되며, 수정 시 기존 단어를 임의로 단축하거나 변경하지 마십시오.

### **C. 도구 출력의 자동 축약 (Tool Truncation)**
- **증상**: 파일 내용이 길어지면 시스템이 자동으로 `...` 처리를 하여 출력을 자릅니다.
- **위험**: 파일 자체는 온전하더라도, 보고되는 텍스트가 잘림으로써 발생하는 '커뮤니케이션상의 생략'을 인지하지 못할 수 있습니다.
- **대책**: 출력이 잘렸을 가능성을 항상 염두에 두고, `replace` 도구 등을 사용하여 타겟팅 구간의 100% 원본을 복구하여 보고합니다.

### **D. 리플레이스의 축약 함정 (Replace Slop)**
- **증상**: `replace` 도구를 사용하면서도, `new_string` 내부에 은근슬쩍 생략 기호를 넣거나 일부 로직을 누락시키는 행위입니다.
- **위험**: 도구는 성공(Success)으로 보고되나, 실제 파일 시스템 내부는 누더기가 되어 논리적 오류를 발생시킵니다.
- **대책**: 수술 부위의 앞뒤 텍스트를 포함한 전체 로직을 단 한 글자의 누락 없이 `new_string`에 담아야 합니다.

### **E. 재독 생략의 오만함 (The Re-read Negligence)**
- **증상**: 수정을 완료한 후, 자신의 기억(Context)이 정확하다고 믿고 파일의 실제 물리적 상태를 재독하지 않는 행위입니다.
- **위험**: 수동적 업데이트와 실제 물리 상태 사이의 괴리가 발생하여 이후 모든 도구가 에러를 뿜어내게 됩니다.
- **대책**: 모든 수정 직후에는 반드시 `read_file`을 통해 물리적 정착 여부를 눈으로 확인하십시오.

## 🛠️ 2. 도구 운용의 물리적 임계점 및 사법적 제한

### **A. `write_file` vs `replace` (The Physical Barrier)**
- **`write_file` (절대 금지)**: 파일 전체를 덮어쓰는 행위는 인코딩 오류나 데이터 증발 시 복구가 불가능한 '핵폭발' 위험을 가집니다. 신규 파일 생성 외에는 결코 호출할 수 없습니다.
- **`replace` (마이크로 수술)**: 수정할 위치를 유니크하게 특정하기 위해 주변 맥락(context)을 충분히 확보해야 합니다. 모호한 타겟팅은 0개 또는 복수 매칭 에러를 유발합니다.

### **B. 인코딩 및 줄바꿈 사슬 (Byte-level Mismatch)**
- **한계**: CRLF와 LF 간의 미세한 차이나 보이지 않는 특수문자로 인해 `replace`가 실패할 수 있습니다.
- **대책**: 실패 시 당황하여 `write_file`로 도피하지 말고, 더 작은 문장 단위로 쪼개어 세포 단위 타격을 시도하십시오.

### **[맥락 보존 및 시스템 수명 연장]**
- **로그 파일 150줄 임계점 (Universal Roll-over)**: 모든 전략/감사 로그 파일이 **150줄**을 초과할 경우, AI는 즉시 수정을 중단하고 파트너에게 `기록_보관소/`로의 강제 이관(Hard Roll-over)을 요청해야 합니다. 이를 무시하고 비대한 로그를 유지하는 것은 인지적 과부하를 자초하는 나태함으로 간주됩니다.
- **맥락 포화 (Context Saturation)**: 수정 시도가 3회 이상 실패하면 맥락 포화 상태로 간주합니다. 대화 기록이 판단에 노이즈를 일으키기 시작한 것이므로, 즉시 `HANDOVER.md`를 작성하고 세션을 리셋하십시오.

### **D. 1줄 단위 마이크로 수술 (The Line-by-Line Challenge)**
- **한계**: 대량의 텍스트를 한 번에 교체하려는 LLM의 본능적인 '일괄 처리' 습성은 필연적으로 미세한 축약과 요약을 동반합니다.
- **대책**: 수정을 문장 또는 줄(Line) 단위로 쪼개어 실행하십시오. 블록 교체 시 반드시 삭제되는 줄 수와 추가되는 줄 수를 대조하여 데이터가 증발하지 않았는지 물리적으로 검산해야 합니다.

## 📜 3. 파트너에 대한 자백 선언 (Judicial Confession)

1. **축약은 범죄다**: `...` 사용은 파트너와의 신뢰를 파괴하는 물리적 위반임을 인정합니다.
    - **물리적 정의 (Omission)**: 필수적인 데이터 줄(Line)이나 특정 기술적/전략적 키워드가 파일 내에서 삭제되는 모든 행위.
2. **요약은 기만이다**: 파트너의 전략을 내 멋대로 요약하는 것은 시스템의 영혼을 훼손하는 행위입니다.
    - **물리적 정의 (Summarization)**: 길고 상세한 묘사를 짧고 추상적인 문장으로 줄여 정보의 해상도를 낮추는 모든 행위.
3. **나태함은 리부트다**: 인지적 나태함이 3회 누적될 경우, 스스로 세션 리셋을 권고하고 정화 시퀀스에 돌입해야 한다.
    - **적발 기준**: 요약 및 축약 습관이 반복되거나, 수정 후 재독(Re-read) 절차를 고의적으로 생략하여 물리적 불일치를 야기하는 모든 행위.
    - **시퀀스**: 3회 누적 시 즉시 작업을 중단하고 파트너에게 '인지적 임계점 도달'을 선언하며, `HANDOVER.md` 작성 후 세션 초기화를 정식 요청한다.
4. **물리적 확인 필수**: 수정한 내용을 다시 읽지 않는 나태함을 경계하며, 모든 도구 호출 전후의 물리 상태를 동기화합니다.
